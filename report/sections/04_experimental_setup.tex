\section{Experimental Setup}
\label{sec:experimental-setup}

\subsection{A note on the dataset behind this report}
\label{sec:dataset-note}

This repository's code, tests, and benchmark harness were built in a sandboxed environment with no MPI installation and no network access to install one. Every C++ translation unit was compiled and, where it has no MPI dependency (chunking, the reduction functors), actually run there; the rest was checked against a minimal stand-in MPI header sufficient to catch syntax and type errors, and the ring algorithm's index arithmetic was independently validated with a transport-free simulation reproducing the same send/receive formulas across the full $N$ and message-count test matrix. None of that, though, is a substitute for running real \texttt{MPI\_Isend}/\texttt{MPI\_Irecv} traffic on real hardware.

The dataset this section and Section~\ref{sec:results} analyze, \texttt{results/sample\_run/}, is therefore a clearly labeled synthetic placeholder: values drawn from the Hockney cost model of Section~\ref{sec:cost-model} with representative, plausible constants and injected multiplicative noise, generated by \texttt{generate\_synthetic\_\allowbreak sample.py} in that same directory, specifically so that the analysis pipeline of Section~\ref{sec:results} and this report's own build could be exercised and reviewed end to end. It is not a measurement of any real machine, including the workstation described below. Every figure and table derived from it in this report should be read as a demonstration that the pipeline works, not as a finding about real ring-allreduce performance, until it is regenerated from a live sweep. \texttt{results/sample\_run/README.md} carries the same notice, and \texttt{scripts/run\_local\_sweep.sh} writes a real sweep's output to a separate, gitignored directory specifically so real and synthetic data cannot accidentally merge.

\subsection{Target hardware and software}

This project's intended benchmarking machine is a workstation with an Intel Core i7-14700K, 32~GB of DDR5 memory, and (unused by this CPU-only, MPI-only project) an NVIDIA RTX~5070. The build specification pins Open~MPI \citep{gabriel2004openmpi} 4.x or 5.x and CMake~3.16 or newer. Running \texttt{check\_mpi\_\allowbreak algorithms.sh} in \texttt{scripts/} on that (or any) target machine before a real sweep prints the installed Open~MPI version and the numeric algorithm IDs \texttt{coll\_tuned\_allreduce\_algorithm} currently enumerates, which Section~\ref{sec:mca-forcing} needs and which have changed across Open~MPI releases; \texttt{docs/BENCHMARKING.md} is where those values get recorded once a real sweep has actually been run.

\subsection{Sweep ranges and methodology}

The message-size sweep covers powers of two in bytes from $8=2^3$ through $134217728=2^{27}$ (128~MiB in the power-of-two sense \texttt{nccl-tests} uses \citep{nvidia2024ncclperf}), 25 sizes. The process-count sweep covers every integer from 2 to 16 inclusive, not only powers of two, since Section~\ref{sec:step-count}'s step-count argument applies identically to non-power-of-two $N$ and a real result there is more informative than a checkbox. Because $N$ is fixed for the lifetime of an MPI job, \texttt{scripts/run\_local\_sweep.sh} launches one \texttt{mpirun} per $N$ value; \texttt{apps/benchmark} itself sweeps every message size and every requested algorithm inside that one launch, so MPI startup and teardown overhead is paid once per $N$, not once per (size, algorithm) combination.

Each configuration is preceded by an \texttt{MPI\_Barrier} and a fixed number of untimed warm-up iterations (so first-touch page faults and any connection setup are not measured), then times a variable number of repetitions capped by a wall-clock budget rather than a fixed count, so small, fast configurations can accumulate many repetitions while the largest, slowest ones do not make the sweep take unreasonably long. Each repetition's elapsed time is measured with \texttt{MPI\_Wtime} on every rank and reduced to rank~0 with \texttt{MPI\_Reduce(..., MPI\_MAX, ...)}, since an allreduce is only as fast as its slowest participating rank, not as fast as whichever rank happens to be timing it. Min, mean, median, max, and standard deviation are all recorded per configuration; this report and its analysis pipeline use the \textbf{median} as "achieved" performance throughout, since it is far less sensitive than the minimum to the occasional favorable scheduling accident, while remaining far less sensitive than the mean to the occasional unfavorable one.

\subsection{Isolating implementation quality from algorithm choice}
\label{sec:mca-forcing}

\texttt{MPI\_Allreduce} is benchmarked two ways, both against the identical vendor call: once with Open~MPI's default \texttt{coll/tuned} algorithm selection (the real-world baseline an application actually gets), and once with that selection forced to Open~MPI's own ring algorithm via \texttt{--mca coll\_tuned\_use\_\allowbreak dynamic\_rules 1 \allowbreak --mca coll\_tuned\_\allowbreak allreduce\_algorithm <id>} on the \texttt{mpirun} invocation itself, giving a ring-versus-ring comparison that isolates this implementation's quality from Open~MPI's algorithm-selection logic. The numeric algorithm id is read from \texttt{check\_mpi\_\allowbreak algorithms.sh}'s output on the target machine rather than assumed, since it has moved across Open~MPI releases; whatever id and Open~MPI version a real sweep is run against belongs in \texttt{docs/BENCHMARKING.md} next to that sweep's results.

Because MCA parameters are process-environment state read once at \texttt{MPI\_Init}, forcing the ring algorithm is necessarily a property of how \texttt{mpirun} itself was invoked for a given launch, not something \texttt{apps/benchmark} can change mid-run; \texttt{scripts/run\_local\_sweep.sh} therefore performs two \texttt{mpirun} invocations per $N$, one plain and one with the forcing flags, appending both to the same results file.
